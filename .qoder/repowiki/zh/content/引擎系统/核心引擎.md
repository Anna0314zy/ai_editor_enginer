## 简介

核心引擎是AI课件编辑器的中央控制单元，负责协调场景管理、历史记录和时间轴播放等功能。该引擎采用命令模式设计，确保所有状态变更都通过统一的执行管道进行，从而保证数据一致性和可撤销性。

引擎的核心设计理念包括：
- **单一职责原则**：每个子系统专注于特定功能领域
- **命令模式**：所有状态变更都封装为可撤销的命令对象
- **分层架构**：清晰分离业务逻辑和表现层
- **框架无关性**：核心引擎不依赖任何特定UI框架
- **事件驱动架构**：支持细粒度的主题化订阅机制

## 详细组件分析

### Engine 类详细分析

#### 核心方法实现

**execute 方法执行机制**：
- 接收 Command 对象作为参数
- 调用命令的 execute() 方法执行实际操作
- 将命令推入历史记录栈以便后续撤销
- 通过 notify('scene') 触发场景主题的订阅通知

**撤销/重做功能调用流程**：
- undo() 方法从撤销栈弹出最近的命令并执行其 undo() 方法
- redo() 方法从重做栈弹出命令并重新执行其 execute() 方法
- canUndo() 和 canRedo() 提供状态查询能力
- 撤销/重做操作会触发场景和历史主题的订阅通知

**编辑器状态管理**：
- getEditorState() 返回当前编辑器状态的副本
- setEditorState() 支持部分状态更新，保持其他属性不变
- 状态更新会触发 editorState 主题的订阅通知

**主题化订阅机制**：
- subscribe() 方法支持两种重载：全局订阅和主题订阅
- unsubscribe() 方法用于取消订阅
- notify() 私有方法负责触发相应的通知

## 性能考虑

### 内存管理

- 使用不可变更新策略避免意外的状态共享
- 命令对象在历史栈中保持引用，便于撤销操作
- 时间轴使用requestAnimationFrame优化渲染性能
- 订阅者集合使用 Set 数据结构，提供高效的添加和删除操作

### 并发处理

- 命令执行采用同步模式，确保操作原子性
- 撤销/重做操作按历史顺序执行
- 时间轴播放使用单线程模型避免竞态条件
- 主题化通知采用同步触发，确保订阅者的及时响应

### 主题化订阅性能优化

- 主题监听器使用 Map 结构存储，按主题类型快速查找
- 全局监听器和主题监听器分离存储，避免不必要的遍历
- 通知机制支持批量触发，减少重复的回调执行

### 最佳实践建议

1. **命令设计原则**：
   - 每个命令应该封装单一的业务操作
   - 命令必须实现完整的撤销逻辑
   - 避免在命令中直接修改全局状态

2. **状态管理**：
   - 使用部分更新避免不必要的状态重建
   - 编辑器状态与场景状态分离管理
   - 合理使用缓存减少重复计算

3. **主题化订阅**：
   - 优先使用主题化订阅而非全局订阅，减少不必要的重渲染
   - 在组件卸载时及时取消订阅，避免内存泄漏
   - 合理组织订阅逻辑，避免重复订阅相同主题

4. **性能优化**：
   - 大量元素操作时考虑批处理
   - 合理设置时间轴刷新频率
   - 使用防抖技术优化频繁操作
   - 利用 React Hooks 的 memoization 优化渲染

## 故障排除指南

### 常见问题及解决方案

**问题1：命令执行后状态未更新**
- 检查命令是否正确实现 execute() 和 undo() 方法
- 确认命令对象是否通过 Engine.execute() 执行
- 验证历史记录栈是否正常工作

**问题2：撤销操作失败**
- 检查命令对象是否正确保存了撤销状态
- 确认历史记录栈中是否存在对应命令
- 验证命令的 undo() 方法实现是否完整

**问题3：时间轴播放异常**
- 检查时间轴状态是否正确初始化
- 确认requestAnimationFrame回调是否正常执行
- 验证播放状态切换逻辑

**问题4：主题化订阅无效**
- 检查主题名称是否正确（'scene' | 'editorState' | 'history' | 'all'）
- 确认订阅回调函数是否正确注册
- 验证引擎状态变更是否触发了相应的通知

**问题5：React Hooks 订阅泄漏**
- 检查组件卸载时是否正确调用了 unsubscribe
- 确认 useEffect 清理函数是否正确返回
- 验证订阅回调函数的引用稳定性
